# 테이블오더 서비스 - 요구사항 정의서

## Intent Analysis

- **User Request**: 테이블오더 서비스 Backend API 구축
- **Request Type**: New Project (Greenfield)
- **Scope Estimate**: System-wide (멀티 매장 지원, 고객/관리자 기능 전체)
- **Complexity Estimate**: Complex (다수의 도메인, 세션 관리)
- **Requirements Depth**: Comprehensive

## Functional Requirements

### FR-1: 매장 관리 (Store Management)

- **FR-1.1**: 멀티 매장 지원 - 여러 매장을 독립적으로 관리
- **FR-1.2**: 매장별 고유 식별자 (storeId) 기반 데이터 격리
- **FR-1.3**: 관리자 계정은 사전 등록 방식 (DB seed)

### FR-2: 인증 및 세션 관리 (Authentication & Session)

#### FR-2.1: 관리자 인증
- 매장 식별자 + 사용자명 + 비밀번호 기반 로그인
- JWT 토큰 발급 (16시간 유효)
- 비밀번호 bcrypt 해싱
- 로그인 시도 제한

#### FR-2.2: 고객 접근 (인증 없음)
- QR코드 스캔으로 매장/테이블 정보가 포함된 모바일 웹 URL 접근
- 별도 로그인 없이 즉시 메뉴 화면 진입
- URL에 매장ID, 테이블 번호 포함 (예: /store/{storeId}/table/{tableNumber})

### FR-3: 메뉴 관리 (Menu Management)

- **FR-3.1**: 메뉴 CRUD (등록, 조회, 수정, 삭제)
- **FR-3.2**: 카테고리별 메뉴 분류
- **FR-3.3**: 메뉴 정보: 메뉴명, 가격, 설명, 카테고리, 이미지
- **FR-3.4**: 메뉴 노출 순서 조정
- **FR-3.5**: 이미지 파일 업로드 (서버 로컬 파일 시스템 저장)
- **FR-3.6**: 필수 필드 및 가격 범위 검증

### FR-4: 주문 처리 (Order Processing)

- **FR-4.1**: 주문 생성 - 매장ID, 테이블번호, 메뉴 목록(메뉴명, 수량, 단가), 총 금액 포함. 세션ID는 서버에서 매장ID+테이블번호 기반으로 현재 활성 세션을 자동 조회하여 연결.
- **FR-4.2**: 주문 상태 관리 - 대기중(PENDING) / 승인(ACCEPTED) / 거절(REJECTED) / 준비중(PREPARING) / 완료(COMPLETED)
- **FR-4.3**: 주문 승인/거절 API (관리자용) - 신규 주문에 대해 관리자가 승인 또는 거절. 거절 시 선택적 거절 사유 입력 가능.
- **FR-4.4**: 주문 상태 변경 API (관리자용) - 승인된 주문의 상태를 준비중/완료로 변경
- **FR-4.5**: 주문 삭제 API (관리자 직권 수정) - 승인/거절 여부와 무관하게 관리자가 주문 자체를 삭제. 삭제된 주문은 총 주문액에서 제외.
- **FR-4.6**: 테이블별 현재 세션 주문 내역 조회
- **FR-4.7**: 주문 시간 순 정렬
- **FR-4.8**: 총 주문액 계산 규칙 - 승인(ACCEPTED), 준비중(PREPARING), 완료(COMPLETED) 상태의 주문만 합산. 대기중(PENDING), 거절(REJECTED) 주문은 제외.

### FR-5: 테이블 세션 관리 (Table Session)

- **FR-5.1**: 테이블 세션 시작 - 해당 테이블의 첫 주문 시 서버에서 매장ID+테이블번호 기반으로 자동 생성. 고객은 세션을 인지하지 않음.
- **FR-5.2**: 테이블 세션 종료 (이용 완료) - 관리자가 수동 처리
- **FR-5.3**: 세션 종료 시 주문 내역을 과거 이력(OrderHistory)으로 이동
- **FR-5.4**: 세션 종료 시 테이블 현재 주문 목록 및 총 주문액 리셋
- **FR-5.5**: 새 고객은 이전 주문 내역 없이 시작

### FR-6: 주문 모니터링 (Order Monitoring)

- **FR-6.1**: 관리자용 주문 목록 조회 API (폴링 방식)
- **FR-6.2**: 매장별 주문 필터링 (storeId 기반)
- **FR-6.3**: 주문 상태별 필터링

### FR-7: 과거 주문 내역 (Order History)

- **FR-7.1**: 테이블별 과거 주문 목록 조회 (시간 역순)
- **FR-7.2**: 주문 정보: 주문 번호, 시각, 메뉴 목록, 총 금액, 이용 완료 시각
- **FR-7.3**: 날짜 필터링 기능

## Non-Functional Requirements

### NFR-1: 성능
- API 응답 시간: 일반 API 500ms 이내

### NFR-2: 보안
- 비밀번호 bcrypt 해싱
- JWT 토큰 기반 인증 (16시간 만료, 관리자만)
- 로그인 시도 제한
- 매장별 데이터 격리 (storeId 기반)
- QR코드 접근 보안: 유효한 매장ID+테이블번호 조합만 허용, 존재하지 않는 조합은 에러 반환

### NFR-3: 데이터 무결성
- 주문 생성 시 트랜잭션 처리
- 세션 종료 시 주문 이력 이동 트랜잭션 처리
- 필수 필드 검증

### NFR-4: API 문서화
- Swagger (OpenAPI 3.0) 기반 API 문서 자동 생성
- 모든 API endpoint 문서화

- 결제 처리 (PG사 연동, 영수증, 환불, 포인트/쿠폰)
- 복잡한 인증 (OAuth, SNS 로그인, 2FA)
- 이미지 리사이징/최적화
- 알림 시스템 (푸시, SMS, 이메일)
- 주방 기능 (주방 전달, 재고 관리)
- 고급 기능 (분석, 매출 리포트, 직원 관리, 예약, 리뷰, 다국어)
- 외부 연동 (배달 플랫폼, POS, 소셜 미디어)
- Frontend 구현 (별도 프로젝트)

## MVP 범위 요약

### 고객용 API
1. QR코드 기반 메뉴 화면 접근 (인증 없음)
2. 메뉴 조회 (카테고리별)
3. 주문 생성
4. 현재 세션 주문 내역 조회

### 관리자용 API
1. 관리자 로그인
2. 메뉴 CRUD + 이미지 업로드
3. 주문 모니터링 (폴링 방식)
4. 주문 승인/거절
5. 주문 상태 변경
6. 주문 삭제
7. 테이블 세션 관리 (이용 완료 처리)
8. 과거 주문 내역 조회
